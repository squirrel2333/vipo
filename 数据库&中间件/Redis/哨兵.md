sentinel

* 主从复制中，主机down，从机原地待命，等待主机重连
* 哨兵巡查监控后台master主机是否故障，如果故障根据投票数自带将某一个库转变为主库，继续服务
* 哨兵自动监控和维护集群，不存放数据
* 三个哨兵组成集群，避免哨兵单点故障，奇数保证投票数

# 配置

sentinel.conf 文件名不能错

* 哨兵默认端口26379

```
bind 0.0.0.0
daemonize yes
protected-mode no
port 26379
logfile "/myredis/sentinel26379.log"
pidfile /var/run/redis-sentinel26379.pid
dir /myredis
sentinel monitor mymaster 192.168.111.169 6379 2
sentinel auth-pass mymaster 111111

sentinel monitor <master-name> <ip> <redis-port> <quorum> # quorum表示最少几票哨兵同意故障迁移

sentinel auth-pass <master-name> <password> # master设置了密码时，连接master的密码

sentinel down-after-milliseconds <master-name> <milliseconds> # 指定多少毫秒之后，主节点没有应答哨兵，此时哨兵主观上认为主节点下线

sentinel parallel-syncs <master-name> <nums> #表示允许并行同步的slave个数，当Master挂了后，哨兵会选出新的Master，此时，剩余的slave会向新的master发起同步数据

sentinel failover-timeout <master-name> <milliseconds> #故障转移的超时时间，进行故障转移时，如果超过设置的毫秒，表示故障转移失败

sentinel notification-script <master-name> <script-path> #配置当某一事件发生时所需要执行的脚本

sentinel client-reconfig-script <master-name> <script-path>#客户端重新配置主节点参数脚本
```
* 为每个哨兵节点创建一份配置

# 启动

```
redis-sentinel sentinel26379.conf --sentinel
reids-sentinel sentinel26380.conf --sentinel
redis-sentinel sentinel26381.conf --sentinel
```
# shutdown主机(6379)

1. 从机数据保留
2. 选举出新的master（6381）
3. 下线的master重连，会被变成slave
* 6379主机下线重连可能会变为从机，如果配置文件中没有配置masterauth，连接会报错
4. 6379自身的redis.conf会被自动修改，加入`replicaof`新的主机
5. 6381的redis.conf也会被修改，删掉`replicaof`

结论：文件内容在运行期间会被sentinel动态修改。master-slave切换后，redis.conf, sentinel.conf都会改变。redis.conf中主从信息改变，sentinel.conf监控目标改变

# 哨兵运行流程/选举原理

## SDown主观下线

单个sentinel主观检测到在给定毫秒数内master状态不可用（心跳失败），就达到SDown条件

## ODown客观下线

哨兵投票达到票数，达成一致，认为master已经down掉

## 选举出leader哨兵

当master节点被判断为客观下线，哨兵节点进行协商，选举出leader哨兵，并由leader哨兵来进行failover（故障迁移）

1. leader哨兵由几个哨兵通过Raft（先到先得）算法选出
2. **leader哨兵从a-c顺序判断选择新的master（在剩余slave节点健康的前提下），如果相同，则通过下一条判断**
1. redis.conf中，优先级slave-priority/replica-priority最高的从节点（数字越小越高）
2. 复制偏移量offset位置最大的节点，即复制最多和原先master最接近的节点
3. 最小Run ID的从节点，ASCII码对照

## 节点更替

1. sentinel leader会对选举出来的新master 执行 slaveof no one 让选出来的从节点成为新的主节点
2. sentinel leader通过向其他slave发送slaveof命令让其它节点成为新master的从节点
3. 下线的老master重新上线，sentinel leader会让原来的master降级为slave并恢复工作

# 建议

1. 哨兵应有多个组成集群
2. 数量为奇数
3. 哨兵节点配置一致
4. 哨兵在docker中注意端口映射
5. **哨兵集群+主从复制不能保证数据零丢失。在选举期间不可用**
