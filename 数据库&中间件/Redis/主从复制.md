
master以写为主，slaver以读为主。当master数据变化时，自动将新的数据异步同步

读写分离、容灾恢复、数据备份、水平扩容支撑高并发

master如果配置了**requirepass**参数，需要密码登录，那么slave就要配置**masterauth**来设置校验密码，否则的话master会拒绝slave的访问请求

# 基本配置命令

```
info relpication # 可以查看复制节点主从关系和配置信息

replicaof 主库ip 主库端口 # 配置文件的配置项

slaveof 主库ip 主库端口 # 命令 运行期间修改slave节点信息，会停止与原主库的同步关系转而和新的主库同步

slaveof no one # 使当前数据库停止与其它数据库同步，转成主数据库
```

# 一主两从

1. 从机只能读取不能写操作
2. slave加入，从头开始复制，后续跟随mater
3. 主机down掉，从机保持数据并等待
4. 主机重启，从机连接，关系照旧
5. 如果从机不是通过配置文件加入主机，而是通过`slaveof`命令，重启之后主从关系会丢失

# 链式

![](主从复制_003.png)

上一个slave使下一个slave的master，slave可以接收其它slave的连接和同步请求，那么slave作为链条中下一个的master，可以有效减轻主master的写压力

* 中途变更会清除之前的数据，重新建立拷贝最新的
* 6380节点作为6381的master，但仍不能进行写操作

# 独立

`slaveof no one`使当前数据库脱离主从关系，变为独自的master，并且数据会保留

# 复制原理

1. slave启动，同步初请
1. slave成功连接到master会发送一个sync命令
2. 首次全新连接master，一次完全同步被执行，slave自身原有数据会被master数据覆盖
2. 首次连接，全量复制
1. master收到sync命令后会开始在后台保存快照(RDB持久化），同时收集所有接收到的用于修改数据命令缓存起来，master执行完RDB后，将rdb快照和所有缓存命令发送到所有slave
2. slave在接收到之后，将其存盘并加载到内存，完成初始化
3. 心跳持续，保持通信
1. master发出ping，默认十秒 `repl-ping-replica-period`
4. 进入平稳，增量复制
5. 从机下线，重连续传
1. master检查backlog里的offset，master和slave会保存一个复制的offset和masterId。master只会把已经复制的offset后面的数据复制给slave，类似断点续传

# 缺点

1. 复制延时，信号衰减。系统繁忙或slave数量过高，问题严重
2. 默认情况，master挂了，slave中不会竞选出新master
