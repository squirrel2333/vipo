
# 设计模式的目的

软件编写过程中，耦合性、内聚性、可维护性、可扩展性、重用性，灵活性等方面的挑战。设计模式为了让程序拥有更好的

1. 代码重用性（相同功能代码不用多次重写）
2. 可读性（编程规范性，便于他人理解）
3. 可扩展性可维护性（当需要增加新功能时成本很低）
4. 可靠性（当增加新功能后对原有功能没有影响）
5. 高内聚，低耦合的特性（模块内部高内聚，模块之间低耦合）

# 设计模式类型

1. 创建型模式：单例、抽象工厂、原型、建造者、工厂
2. 结构型模式：适配器、桥接、装饰、组合、外观、享元、代理
3. 行为型模式：模板方法、命令、访问者、迭代器、观察者、中介者、备忘录、解释器、状态、策略、职责链

# 设计模式七大原则

## 单一职责原则

### 基本介绍

对类来说，即一个类只负责一个职责

### 案例

```
public class SingleResponsibility1 {
    public static void main(String[] args) {
        Vehicle vehicle = new Vehicle();
        vehicle.run("摩托车");
        vehicle.run("汽车");
        vehicle.run("飞机");
    }
}

/*
方式1：
1.在方式中run方法违反了单一职责原则
2.解决方案是依据不同的交通工具运行方法，分解成不同的类
 */
class Vehicle {
    public void run(String vehicle) {
        System.out.println(vehicle + "在公路上运行...");
    }
}

/*
方案2：
1. 遵守单一职责原则
2. 但改动大，同时得修改客户端
3. 通过方案3改进
 */
class RoadVehicle {
    public void run(String vehicle) {
        System.out.println(vehicle + "在地上跑");
    }
}

class AirVehicle {
    public void run(String vehicle) {
        System.out.println(vehicle + "在空中飞");
    }
}

class WaterVehicle {
    public void run(String vehicle) {
        System.out.println(vehicle + "在水里跑");
    }
}

/*
方式3：
1.这种修改没有对原来的类做大修改，只增加方法
2.虽然没有在类的级别遵守单一职责原则，但是在方法上遵守了
 */
class Vehicle {
    public void run(String vehicle) {
        System.out.println(vehicle + "在公路上运行...");
    }

    public void runAir(String vehicle) {
    }

    public void runWater(String vehicle) {
    }
}
```
### 注意事项

1. 降低类的复杂度，一个类只负责一项职责 。
2. 提高可读性，可维护性
3. 降低变更引起的风险（例如在一个类里加判断）
4. 通常应该遵守单一职责原则；只有在逻辑足够简单时，才可以在代码级别违法单一职责原则；类中方法足够少时，可以在方法级别保持单一职责原则

## 依赖倒置原则

### 基本介绍

1. 高层模块不应该依赖低层模块，二者应该依赖抽象
2. 抽象不应该依赖细节，细节应该依赖抽象
3. 依赖倒置的中心思想是面向接口编程
4. 相对于细节的多样性，抽象的东西应该要稳定。在Java中抽象指的是接口或者抽象类，细节就是具体类
5. 接口和抽象类目的是制定规范，而不涉及具体操作

### 案例

```
public class DependencyInversion {
    public static void main(String[] args) {
        Person person = new Person();
        person.receive(new Email());
    }
}

class Email {
    public String getInfo() {
        return "hello, world";
    }
}

//Person 接收消息
/*
    方式1：
    1. 简单
    2. 如果我们获取对象是微信类，Person也要增加相应的接受方法
    3. 解决：引入一个抽象的接口，表示接收者，Person类与接口发生依赖
        因为Email、Wechat等等属于接收范围，各自实现receiver接口就行，符合依赖倒置原则
 */
class Person {
    public void receive(Email email) {
        System.out.println(email.getInfo());
    }
}
```
依赖倒置原则优化：

```
public class DependencyInversion {
    public static void main(String[] args) {
        Person person = new Person();
        person.receive(new Email()); //无需改变
        person.receive(new Wechat());
    }
}

//定义接口
interface IReceiver {
	String getInfo();
}

class Email implements IReceiver{
    @Override
    public String getInfo() {
        return "电子邮件：hello, world";
    }
}

class Wechat implements IReceiver{
    @Override
    public String getInfo() {
        return "Wechat：hello, world";
    }
}
//Person 接收消息
/*
	方式二
*/

    }
}

### 注意事项

1. 底层模块尽可能都要有抽象类或者接口，程序稳定性会更好
2. 变量的声明类型尽量是抽象类或者接口， 这样引用变量和实际对象之间，就存在缓冲层，利于程序扩展和优化。
3. 继承时遵循里氏替换原则。

## 里氏替换原则

### 继承思想的思考

1. 父类中已经实现好的方法，实际上是设定规范和契约。虽然不强制遵循这些契约，但如果子类对这些已经实现的方法任意修改，就会对整个继承体系造成破坏。
2. 继承会给程序带来入侵性，程序的可移植性下降，增加对象间的耦合性，如果一个类被其它类所继承，则这个类修改时，必须考虑到所有子类
3. 里氏替换原则时考虑如何正确使用继承

### 基本介绍

概括为：引用基类的地方必须能透明的使用其子类对象

尽量不要重写父类的方法

继承实际上让两个类耦合性增强了，适当情况下，可以通过**聚合，组合，依赖**来解决问题

### 案例

```
public class LiskovDemo {
    public static void main(String[] args) {
        A a = new A();
        System.out.println("11 - 1 = " + a.func1(11, 1));
        B b = new B();
        System.out.println("11 -3 " + b.func1(11, 3)); //本意求出11-3
        System.out.println("11 + 3 + 9" + b.func2(11, 3));
    }
}

class A {
    public int func1 (int num1, int num2) {
        return num1 - num2;
    }
}

class B extends A {
    // 无意识的重写
    public int func1 (int num1, int num2) {
        return num1 + num2;
    }
    public int func2 (int num1, int num2) {
        return func1(num1, num2) + 9;
    }
}
```
里氏替换原则改进：

```
public class LiskovDemo {
    public static void main(String[] args) {
        A a = new A();
        System.out.println("11 - 1 = " + a.func1(11, 1));
        B b = new B();

        System.out.println("11 + 3 " + b.func1(11, 3));
        System.out.println("11 + 3 + 9" + b.func2(11, 3));
        System.out.println("11 - 3" + b.fun3(11, 3));

## 开闭原则

### 基本介绍

1. 是最基础、最重要的设计原则
2. 软件实体，模块和函数应该对扩展开放（对提供方），对修改关闭（使用方）。
3. 当软件需要变化时，尽量通过扩展软件实体的行为实现变化，而不是通过修改已有代码来实现。

### 案例

```
public class Opc {
    public static void main(String[] args) {
        Shape circle = new Circle();
        Shape rectangle = new Rectangle();
        GraphicEditor ge = new GraphicEditor();
        ge.drawShape(circle);
        ge.drawShape(rectangle);

    }
}

class GraphicEditor {
    public void drawShape(Shape s) {
        if (s.m_type == 1)
            drawRectangle(s);
        else if (s.m_type == 2)
            drawCircle(s);
    }

    private void drawRectangle(Shape r) {
        System.out.println("绘制矩形");
    }

    private void drawCircle(Shape r) {
        System.out.println("绘制圆形");
    }
}

class Shape {
    int m_type;
}

class Rectangle extends Shape {
    public Rectangle() {
        super.m_type = 1;
    }
}

class Circle extends Shape {
    public Circle() {
        super.m_type = 2;
    }
}
```
上例中如果要增加三角形，那么使用方（GraphicEditor）也要修改，违反了OCP原则

```

    }

abstract class Shape {

}

```

## 迪米特法则

### 基本介绍

1. 对象应该对其它对象保持最少的了解
2. 类与类关系越密切，耦合度越大
3. 对于被依赖的类不管多复杂，都尽量将逻辑封装在类内部。多外除了提高public方法，不泄露信息
4. 只与直接的朋友通信
5. 直接的朋友：每个对象都会与其它对象有耦合关系。耦合的方式很多：依赖、关联、组合、聚合等。其中，我们称出现的成员变量，方法参数，方法返回值中的类为**直接的朋友**，而出现在局部变量中的类**不是直接朋友**。也就是说，陌生的类不要以局部变量的形式出现在类的内部

## 合成复用原则

尽可能使用合成/聚合的方式，而不使用继承

如果我们只是让B类去使用A类 的方法，继承就会让B和A的耦合性增强

## 设计原则核心思想

1. 找出可能需要的变化之处，把它独立出来，不要和那些不需要变化的代码混合在一起。
2. 针对接口编程，而不针对实现编程
3. 为了交互对象之间的松耦合设计而努力
